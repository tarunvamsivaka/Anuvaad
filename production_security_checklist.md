# Production Security Auditing & Secure Coding Checklist

This document provides a comprehensive, industry-standard checklist and framework for auditing production applications. It covers common vulnerability classes, theoretical attack mechanics, secure implementation patterns, and strategic recommendations for hardening production systems.

---

## 1. Authentication & Session Management

### Vulnerability Class: Weak Authentication & Session Hijacking
Authentication mechanisms must ensure that user identities are verified securely and sessions are protected from hijacking, fixation, or brute-force attacks.

#### Theoretical Attack Scenarios
*   **Brute-Force & Credential Stuffing:** Attackers use automated tools to test lists of leaked credentials against login endpoints that lack rate limiting or account lockout controls.
*   **Session Token Leakage:** Session identifiers are exposed via URL parameters, insecure cookies (missing `Secure` or `HttpOnly` flags), or browser local storage, allowing attackers to hijack active sessions.

#### Secure Implementation Checklist
- [ ] **Enforce Multi-Factor Authentication (MFA):** Implement MFA for all administrative and user accounts.
- [ ] **Secure Cookie Attributes:** Set `HttpOnly`, `Secure`, and `SameSite=Strict` (or `Lax`) attributes on all session cookies to prevent access via Client-Side Scripting (XSS) and cross-site requests.
- [ ] **Implement Rate Limiting:** Enforce strict rate limits on sensitive endpoints (login, password reset, MFA verification) to prevent brute-force attacks.
- [ ] **Cryptographically Secure Session Tokens:** Use high-entropy, random identifiers generated by cryptographically secure pseudo-random number generators (CSPRNGs).
- [ ] **Session Expiry & Revocation:** Implement both idle and absolute session timeouts, and provide server-side mechanisms to invalidate sessions upon logout.

#### Secure Code Pattern (Conceptual Node.js / Express Cookie Config)
```javascript
// Secure session cookie configuration
app.use(session({
  secret: process.env.SESSION_SECRET,
  name: '__Host-SessionID', // Prevents subdomain cookie hijacking
  resave: false,
  saveUninitialized: false,
  cookie: {
    httpOnly: true, // Prevents client-side scripts from reading the cookie
    secure: true,   // Ensures cookie is only sent over HTTPS
    sameSite: 'strict', // Mitigates CSRF risks
    maxAge: 3600000 // 1 hour expiration
  }
}));
```

## 2. API Security & Access Control

### Vulnerability Class: Broken Object Level Authorization (BOLA) & IDOR
Access control mechanisms must verify that the authenticated user has authorization to perform the requested action on the specific target resource.

#### Theoretical Attack Scenarios
*   **Insecure Direct Object Reference (IDOR):** An attacker manipulates a parameter (such as `user_id` or `invoice_id` in a URL or API request) to access records belonging to other users without authorization.

#### Secure Implementation Checklist
- [ ] **Implement Policy-Based Access Control:** Define clear authorization models (RBAC or ABAC) and enforce them programmatically at the API gateway or controller level.
- [ ] **Use Non-Guessable Identifiers:** Use UUIDv4 or secure hashes instead of auto-incrementing integer IDs for resources exposed to users.
- [ ] **Context-Aware Validation:** Ensure every API endpoint validates that the active session owner owns or has explicit permission to access the requested resource ID.
- [ ] **Minimize Data Exposure:** Avoid returning complete database records in API responses; only return fields required by the UI.

#### Secure Code Pattern (Conceptual Python / FastAPI Access Check)
```python
@app.get("/api/v1/invoices/{invoice_id}")
async def get_invoice(invoice_id: str, current_user: User = Depends(get_current_user)):
    # Retrieve resource
    invoice = db.get_invoice_by_id(invoice_id)
    if not invoice:
        raise HTTPException(status_code=404, detail="Invoice not found")
    
    # Explicit ownership validation
    if invoice.owner_id != current_user.id:
        raise HTTPException(status_code=403, detail="Not authorized to view this resource")
        
    return invoice
```

## 3. Injection Prevention

### Vulnerability Class: SQL Injection (SQLi) & Command Injection
Untrusted input must never be directly concatenated or executed as code or commands by backend interpreters.

#### Theoretical Attack Scenarios
*   **SQL Injection:** An attacker inputs SQL commands into an input field (e.g., `' OR '1'='1`), which are concatenated into a dynamic SQL query, allowing them to bypass authentication or extract database contents.
*   **Command Injection:** An application passes unsanitized user input directly to system shells (e.g., executing `system("ping " + user_input)`), allowing attackers to run arbitrary system-level commands.

#### Secure Implementation Checklist
- [ ] **Use Parameterized Queries / ORMs:** Use prepared statements, parameterized queries, or Object-Relational Mappers (ORMs) to handle database queries.
- [ ] **Strict Input Validation:** Use allow-list validation for all user inputs (validating types, patterns, lengths, and formats).
- [ ] **Avoid Shell Execution:** Do not pass user input directly to system command shells. Use native programming language APIs instead of executing shell utilities.
- [ ] **Context-Aware Encoding:** Encode output based on the rendering context (HTML, JS, CSS) to prevent Cross-Site Scripting (XSS).

#### Secure Code Pattern (Conceptual Python / PostgreSQL Prepared Statement)
```python
# SECURE: Using parameterized query
query = "SELECT * FROM users WHERE email = %s AND status = %s"
cursor.execute(query, (user_email, "active"))
results = cursor.fetchall()
```

## 4. Sensitive Data Exposure & Cryptography

### Vulnerability Class: Insufficient Cryptographic Controls
Sensitive data must be protected both in transit and at rest using strong, modern cryptographic algorithms.

#### Theoretical Attack Scenarios
*   **Cleartext Data Transit:** Data transmitted over unencrypted HTTP can be intercepted by attackers on the network path (Man-in-the-Middle).
*   **Weak Password Hashing:** Storing passwords using fast or obsolete hash algorithms (such as MD5 or SHA1) allows attackers to perform rapid offline decryption if the database is leaked.

#### Secure Implementation Checklist
- [ ] **Enforce TLS / HTTPS:** Configure TLS 1.3 (or TLS 1.2 minimum) for all web traffic and API endpoints, and enforce it using HTTP Strict Transport Security (HSTS).
- [ ] **Strong Password Hashing:** Hash passwords server-side using slow, salt-based algorithms such as Argon2id or bcrypt.
- [ ] **Secure Key Management:** Store cryptographic keys, API tokens, and database credentials in dedicated secret managers rather than environment files or code repositories.
- [ ] **Encrypt Sensitive Data at Rest:** Encrypt highly sensitive database columns (e.g., PII, financial info) using strong symmetric encryption (AES-256-GCM).

#### Secure Code Pattern (Conceptual Python / bcrypt Hashing)
```python
import bcrypt

def hash_password(password: str) -> str:
    # Generate salt and hash password
    salt = bcrypt.gensalt(rounds=12)
    hashed = bcrypt.hashpw(password.encode('utf-8'), salt)
    return hashed.decode('utf-8')

def verify_password(password: str, hashed: str) -> bool:
    # Safely compare hashes
    return bcrypt.checkpw(password.encode('utf-8'), hashed.encode('utf-8'))
```

## 5. Infrastructure & Environment Security

### Vulnerability Class: Misconfigured Containers & Environment Exposure
Cloud infrastructure, container environments, and servers must be hardened against unauthorized access and privilege escalation.

#### Theoretical Attack Scenarios
*   **Exposed Secret Credentials:** Sensitive database passwords or API keys are accidentally committed to source repositories or left in accessible environment files on production servers.
*   **Privileged Containers:** Running Docker containers as the `root` user allows an attacker who exploits an application vulnerability to easily compromise the host system.

#### Secure Implementation Checklist
- [ ] **Principle of Least Privilege (PoLP):** Run container processes as non-root users and grant minimal database permissions required for application function.
- [ ] **Secure Container Images:** Use minimal, verified base images (e.g., alpine or distroless) and regularly scan containers for known CVEs.
- [ ] **Disable Debug Modes:** Ensure debug logs, verbose error traces, and administrative panels are completely disabled in production environments.
- [ ] **Implement Dependency Scanning:** Use automated tools (e.g., `npm audit`, `pip-audit`, Dependabot) to monitor and patch vulnerabilities in third-party dependencies.

#### Secure Dockerfile Pattern (Conceptual Secure Multi-stage Build)
```dockerfile
# Stage 1: Build environment
FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

# Stage 2: Runtime environment
FROM python:3.11-slim
WORKDIR /app

# Create a non-privileged user and group
RUN groupadd -r appgroup && useradd -r -g appgroup appuser

# Copy dependencies and source code
COPY --from=builder /root/.local /home/appuser/.local
COPY . /app

# Set environment paths and ownership
ENV PATH=/home/appuser/.local/bin:$PATH
RUN chown -r appuser:appgroup /app /home/appuser

# Switch to the non-privileged user
USER appuser

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```
